# AI Driven 開發系統

分兩層,避免流於「我用 AI 幫忙寫程式」的空泛口號。

## 1. 產品功能 AI

- 將結構化天氣資料轉成自然語言行前建議(例:「週六午後降雨機率高,建議室內備案 / 攜傘」)。
- 實作於 `backend/app/services/ai_summary.py`:
  - 有 `GEMINI_API_KEY` → 標準 `google-genai` SDK 呼叫 Gemini(**不使用任何本機私有 wrapper**,作品自我完備)。
  - 無 key → 規則式摘要(deterministic),功能永不硬失敗,零憑證可 demo。
  - 呼叫失敗 → 優雅降級回規則式,天氣結果不受影響。

## 2. 開發流程 AI

- 需求 → API 規格草稿、產生測試案例、生 PR 描述、整理變更摘要。
- 本專案實際以**多 AI 角色協作**(規劃 / 實作 / 審查 / PR review)推進——這同時是「多人協作 Git 流程」加分項的真實體現。

### 多 Agent 協作紀錄

脫敏後的原始任務卡、開發日誌與驗收軌跡保留於 [`docs/dev-process/`](dev-process/)。
流程上由管理層負責需求拆解、規劃與派工,編碼層依任務卡實作並提交報告,驗收層再以獨立視角執行檢查與回饋。
這些紀錄補足本文件的流程敘述,可追溯從決策、實作到驗收的分層協作鏈路。

## 資料治理邊界(金融業情境主動說明)

- 送往 Gemini 的**僅為公開天氣 / 景點資料,不含任何個資(PII)**。
- Free tier 內容可能被用於改善產品,僅適合開發 / demo;**若進正式生產,改採 paid tier / enterprise route**(資料不被用於訓練)。
- 金鑰走環境變數(開發)→ Secret Manager(生產),不進 repo、不進前端。

## 為何這樣定位較成熟

- AI 在開發層提高交付效率、在產品層做資訊增值,而非取代核心資料流程。
- 天氣預報等關鍵事實仍由 CWA 權威資料決定,AI 只負責「把資料講成人話」,可稽核、可降級。
